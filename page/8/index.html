Windows命令行

Posted on 2016-07-16 Edited on 2018-08-30

Win+R

  1. cmd => 命令行
  2. lpksetup => 弹出安装或者卸载Windows显示语言
  3. ipconfig => 查看IP配置
  4. dcpromo => Windows Server 安装域控制器
  5. compmgmt.msc => 直接打开计算机管理
  6. msconfig => 打开Windows配置
  7. services.msc => 打开windows服务
  8. control => 打开控制面板
  9. mstsc => 打开远程连接

CD

cd的全称是Change Directory，直译为改变文件夹，也就是跳转目录、切换路径的意思。它后面可以接驱动器符号、完整路径和相对路径。一般我们打开命令行窗口的时候，默认的目录位于当前用户所在的路径下，比如：C:\Documents and Settings\jacran>

  1. 跳转到当前驱动器的根目录 cd [当前驱动器盘符]:\ 例如： cd c:\` 或者更简单的 cd\\
  2. 跳转到当前驱动器的其他文件夹 以C盘下的WINDOWS文件夹为例 输入：cd C:\WINDOWS
  3. 跳转到其他驱动器 以从C盘跳转到D盘为例 在任意目录下直接输入： D:
  4. cd /d e:\software 跳转到其他驱动器的其他文件夹, 从C盘跳转到E盘下边的software文件夹。注意此处必须加/d参数。
  5. cd.. 跳转到上一层目录

Command

  1. Win+R => cmd => net user username /active:no 禁用某个用户名

滴滴打车APP3.0改版有感

Posted on 2016-07-16 Edited on 2020-03-04

三周年了

2015-09-09，公司三周年了，这一天，公司从滴滴打车更名为滴滴出行，预示着公司将承载着更大的梦想。产品首页更是大改，作为一个在滴滴两年的员工，内心还是无比的激动，过去的两年伴随着公司的成长个人各方面能力都有提升，能在这样一家年轻，激情的公司，能和一群有理想的同学们一起，实属荣幸。

作为一个带有极简主义倾向，和轻微强迫症的前FE同学现RD未来的PM同学，针对新首页的设计也想谈谈自己的想法。

个人认为不合理点

  1. 厚重的header很容易让人压抑感，第一次看到设计稿的时候第一感觉就是如此，就好比本来一个身材俊俏的小伙突然变得矮胖矮胖了，而且在切换到顺风车时更是如此。(具体参考下图或者打开app)
  2. header上三个icon看起来很僵硬，特别是通知的icon在切换到顺风车和其他产品线还不一致的时候。
  3. 最左边的icon可以考虑把人头换掉了，用三道杠和三个点吧，毕竟里面的内容并非都是指人，而是诸如设置，消息中心，积分商城，发现等功能。
  4. 底部的出发地和目的地会一直占用很大的屏幕，特别在iPhone4和iPhone5上，感觉整个app没有空间感。
  5. 底部选择时间的交互方式找了很久，既然是低频需求，可以考虑在切换到输入目的地的时候再加上该选项吧。
  6. header第二行右边还有三个点的icon，跟通知的icon太近，而且太靠近右边，整个这一块离简洁太远了。

YY下个人设计的极简版滴滴打车

  1. header保持一贯的清爽。那个小人上边说了可以换一下找一下感觉。
  2. 另外不建议放一个固定的通知按钮。
  3. header正中间的logo其实还可以考虑随着下面业务条的切换而变化。
  4. 下面的业务条保持简洁，没有被选中的icon都用点替代，下面带上文字描述。
  5. 最重要的把中间的定位提示框，加上在哪儿，和去哪儿，如果是顺风车的话切换司机和乘客，在哪儿和去哪儿都可以点击进去更改，其他功能，类似加价选车型还有原来的捎话等功能都可以放到第二个页面中去完成。（当然了这个圆形设计得比较low，就别看啦，哈哈）

顺带说一下功能

  1. 原先的附近有xx辆车此功能应该是当年为了吸引乘客叫车，现在该数值对用户叫车行为应该没有直接影响，用户会凭着个人生活经验以及地图上是否有车的logo来判断此处是否有车，而且说实话大部分人都知道那个车辆数并非准备可信，所以还鸡肋还不如直接下掉，或者找个地方简单展示。

  2. 右上角通知消息固定入口，可以考虑挪到我里边真正的消息处吧，毕竟真的想强推消息的时候，在进入app或者切换到业务线的时候直接展示即可。感兴趣的用户第一次就会点进去，不感兴趣的，即使你摆放一个固定的按钮他好奇点了一下，也不至于会在进入落地页看详情。还不如让真正的消息中心发挥作用。

值得表扬的地方

  1. 新首页终于把出发地和目的地上下排列了，作为打开这个app想叫车的人而言，第一眼我想看我在哪里，第二眼我会点击按钮输入我要去哪里。把他俩放一起更聚焦，而不至于像原来那样我还得着重看一下文字原来这里是输入我想去哪儿的?
  2. 整个色调保持一贯的小清新，很爽。

另外下午看到有微博说视障用户的兼容性问题，确实也应该值得改进。

郑重声明：此文绝非吐槽，仅仅谈谈自己的小想法。设计师PM和程序员都很不容易，应该给予高度尊重！

遇到奇葩问题

router使用HTML5的API时候需要注意nginx和apache配置

在配置SPA路由的时候，如果使用H5的historyAPI方式，当用户进入到任何一个页面需要刷新的时候，如果不做特殊配置就会报404错误，这个时候需要Nginx或者Apache做一些rewrite配置，参考

而实用hash的方式，则不需要额外配置。

举例说明

  1. 前端 /a -> /b 浏览器的地址栏变了，location.search变了，这个时候，点击刷新的时候，请求直接到server上，如果server不做配置，那么很可能直接返回404，解决方案就是在server上做一个路由配置，每次都到入口，入口的前端路由在跳转到/b上
  2. 前端 #!/a -> #!/b 浏览器的地址栏不变，变得是location.hash, 这个时候点击刷新的时候，请求直接还是到入口，入口根据hash直接跳回就行了，不需要做server端配置

libraries和helpers区别 参考文章

  • helper里主要是一些函数，
  • library里主要是一些类的对象。函数表示的是一种行为，类表示的是一种抽象的概念，是一系列属性和方法的集合，对象里的函数称为方法，数据则称为对象的属性。类是有状态的，而函数无状态的，所以函数与类之间最大的区别在于是否有状态，落到更实际的点就是方法与方法之间是否需要共享数据，如果需要共享数据，则可以写成类的方法，如果不需要共享数据，那就可以用写成函数。

我的理解：

  1. libraries是有生命周期的，甚至你可以通俗的认为他是通过class来组织的代码
  2. helpers是没有生命周期的，是一组相似方法的集合。

单一原则

不管函数或者类需要遵守一个很重要的原则：单一职责， - 即一个类，最好只做一件事，只有一个引起它变化的原因。如果一个函数用一句话描述不清楚，那就需要拆成多个函数；如果一个类有跟它不相干的职责，那就拆成多个类 - 函数应该短小精悍，如果一个函数太长，意味着关联太多，可拆分的可能性很大。编码的时候应时刻提醒自己，它是不是只做了一件事，还能不能在细分？单一职责是程序员要遵守的首要原则

constants和config的区别

  1. 常量可以不用理会变量范围的规则而在任何地方定义和访问； 常量一旦定义就不能被重新定义或者取消定义；
  2. config是配置，配置是程序的开关，线上线下的不同点，这些数据就应该写成配置，比如API接口，比如数据库配置。
  3. 常量在ci中应该写在config/constants.php文件中

类常量或者说叫枚举

使用的时候

  1. 这些线上线下不会变化的。
  2. 方便阅读代码避免在代码中直接写1，2，等数字

  2. log_message()函数最终调用的还是ci的 Log.php的write_log()方法
  • load_class()在CI的core/Common.php文件中
