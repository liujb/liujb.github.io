Install package managment

using vundle(在.vimrc中添加)

Type of plugins

  1. 在Github vim-scripts 用户下的repos,只需要写出repos名称
  2. 在Github其他用户下的repos, 需要写出”用户名/repos名”
  3. 不在Github上的插件，需要写出git全路径

Install plugins

打开一个vim, 运行:BundleInstall 或者在命令行运行 vim +BundleInstall +qall

Others command

  1. 更新插件:BundleUpdate
  2. 清除不再使用的插件:BundleClean – BundleClean会清除所有不是用Vundle安装的插件，除了Vundle自己
  3. 列出所有插件:BundleList
  4. 查找插件:BundleSearch

AngularJS应用的解析

  • 模板（Templates） 模板是您用HTML和CSS编写的文件，展现应用的视图。 您可给HTML添加新的元素、属性标记，作为AngularJS编译器的指令。 AngularJS编译器是完全可扩展的，这意味着通过AngularJS您可以在HTML中构建您自己的HTML标记！

  • 应用程序逻辑和行为（Logic and Behavior） 应用程序逻辑和行为是您用JavaScript定义的控制器。AngularJS与标准AJAX应用程序不同，您不需要另外编写侦听器或DOM控制器，因为它们已经内置到AngularJS中了。这些功能使您的应用程序逻辑很容易编写、测试、维护和理解。

  • 模型数据（Data） 模型是从AngularJS作用域对象的属性引申的。模型中的数据可能是Javascript对象、数组或基本类型，这都不重要，重要的是，他们都属于AngularJS作用域对象。 AngularJS通过作用域来保持数据模型与视图界面UI的双向同步。一旦模型状态发生改变，AngularJS会立即刷新反映在视图界面中，反之亦然。

  • 这些服务可以让您非常方便的编写WEB应用。

背景

通过配置DNS欺骗，达到preview环境验证服务的case。直白的说就是浏览器从发起对域名1的请求，而通过DNS欺骗的手段，让请求真实到达域名2所在机器。

假设域名1和域名2为：

  1. gsb.ins.xxx.com
  2. gsb.preview.xxx.com

这种方式还可以欺骗浏览器，在做微信开发的时候经常使用。

现象

代码中访问的是gsb.ins.xxx.com，但是需要将实际请求落到gsb.preview.xxx.com，一直以来都是利用Charles的DNSSpoofing方式进行，配置如图：

配置成功效果如图

可以清晰得看到Remote Address 为我们的目标地址。正常来讲这就可以了，况且在过去很长时间我都是这么玩的。但是实际效果并非如此（通过比对两个服务接口的返回不同）。经过排查得到两个域名对应的IP地址其实是同一个。

  1. gsb.ins.xxx.com ＝》123.151.179.1
  2. gsb.preview.xxx.com ＝》123.151.179.1

所以这样子通过DNS-Spoofing还是访问的同一个IP。因为上游机器在辨别两个域名的时候，是通过nginx rewrite的方式进行的，所以通过域名2请求的服务还是落到了域名1所在的机器，故问题得到查明。

原则上两个域名应对应两个IP

FastCGI

  1. FastCGI，也称为FCGI，是一个外部服务与Web服务器的接口协议。FastCGI是早期的通用网关接口(CGI)的增强版本。 FastCGI致力于减少网页服务器与CGI程序之间的开销，从而使服务器可以同时处理更多的网页请求。

  2. Nginx为了许多外部工具和服务而引入FastCGI技术，例如Perl, PHP 和 Python。所以，在FastCGI服务启动之前，它们都是不能使用的。

  3. 有不同的方式可以启动PHP FastCGI服务,我们将介绍php-fpm.这也是推荐的方案.

Git操作之Fetch和Pull的区别

Posted on 2016-07-16 | Edited on 2018-09-01

Git中从远程的分支获取最新的版本到本地有这样2个命令，fetch 和 pull，对于这两者之间的区别并不熟悉，特此记录。

fetch

相当于是从远程获取最新版本到本地，不会自动merge。

以上命令的含义

  1. 首先从远程的origin的master主分支下载最新的版本到origin/master分支上。
  2. 然后比较本地的master分支和origin/master分支的差别。
  3. 最后本地进行合并。

上述过程其实可以用以下更清晰的方式来进行

从远程获取最新的版本到本地的test分支上之后再进行比较合并

pull

从远程获取最新版本并merge到本地。

命令其实相当于git fetch 和 git merge

Reference

  • 参考地址

在实际使用中，git fetch更安全一些因为在merge前，我们可以查看更新情况，然后再决定是否合并
